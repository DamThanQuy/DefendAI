Dự án DefendAI · Chủ đề: FE đọc role từ BE (đã có sẵn) để phân trang / phân quyền UI theo student / mentor / admin.

Module: Auth / RBAC (FE) FE: apps/web (Next.js 14 App Router) BE: apps/api (FastAPI) Ngày: 2026-07-16 Trạng thái: Plan

Mục tiêu

  • FE biết user đang login thuộc role nào (student / mentor / admin) và dùng để:
    • Guard route: chỉ role được phép mới vào trang tương ứng.
    • Hiển thị UI khác nhau (vd: admin thấy menu quản trị, student không thấy).
  • Không gọi lại BE mỗi lần render — đọc từ localStorage.user.roles (BE đã trả sẵn roles: string[]).
  • Single source of truth: role lấy từ user.roles[0] (primary role) do _make_auth_response sinh.

2. Thiết kế tổng thể

Login BE → localStorage.user.roles = ["student"] → useAuth() hook đọc roles → AuthGate guard route → Navbar / page render theo role
  • Hook tập trung: tạo useAuth() trả { user, roles, isRole } để mọi component dùng chung, tránh đọc localStorage rải rác.
  • Guard: mở rộng AuthGate thành ROLE_ROUTES — map route → danh sách role được phép.
  • UI: Navbar ẩn/hiện link theo role (vd: /admin chỉ admin).

3. Các bước implement

1Tạo hook useAuth Mới

File apps/web/src/hooks/useAuth.ts

  • Đọc localStorage.user, parse an toàn, trả roles: string[] + helper hasRole(...).
  • Lắng nghe event storage (giống Navbar) để cập nhật khi login/logout.
import { useEffect, useState } from "react";

export interface AuthUser {
  email: string;
  full_name?: string | null;
  roles: string[];
}

export function useAuth() {
  const [user, setUser] = useState<AuthUser | null>(null);

  useEffect(() => {
    const read = () => {
      const raw = localStorage.getItem("user");
      if (!localStorage.getItem("token") || !raw) { setUser(null); return; }
      try {
        const u = JSON.parse(raw) as Partial<AuthUser>;
        setUser({ email: u.email ?? "", full_name: u.full_name ?? null, roles: u.roles ?? [] });
      } catch { setUser(null); }
    };
    read();
    window.addEventListener("storage", read);
    return () => window.removeEventListener("storage", read);
  }, []);

  const roles = user?.roles ?? [];
  const hasRole = (...r: string[]) => roles.some((x) => r.includes(x));
  return { user, roles, hasRole, isAuthed: !!user };
}

2Mở rộng AuthGate theo role Sửa

File apps/web/src/components/auth/AuthGate.tsx

  • Giữ guard "có token" cho PROTECTED_PATHS.
  • Thêm ROLE_ROUTES: route → role được phép. Nếu user không có role → redirect / (hoặc /403).
const ROLE_ROUTES: Record<string, string[]> = {
  "/admin": ["admin"],
  // ví dụ: "/mentor-review": ["mentor", "admin"],
};

// trong useEffect, sau khi check token:
for (const [path, allowed] of Object.entries(ROLE_ROUTES)) {
  if (pathname === path || pathname.startsWith(`${path}/`)) {
    const roles = (JSON.parse(localStorage.getItem("user") || "{}").roles) || [];
    if (!roles.some((r: string) => allowed.includes(r))) {
      router.replace("/");
      return;
    }
  }
}
Lưu ý: Guard client-side chỉ là UX. Bảo mật thực sự nằm ở BE (dependency require_roles trong deps.py). FE guard tránh user lạc vào trang không liên quan, không thay thế BE auth.

7. Rủi ro & lưu ý

  • Client guard ≠ bảo mật: user có thể sửa localStorage giả role. BE phải luôn check lại qua require_roles trong deps.py cho mọi API nhạy cảm.
  • Multi-role: thiết kế lấy roles[0] làm primary. Nếu sau này 1 user nhiều role, hasRole đã hỗ trợ OR — chỉ cần mở rộng ROLE_ROUTES.
  • SSR mismatch: useAuth đọc localStorage → chỉ chạy client ("use client"). Không dùng trong Server Component.
  • Stale data: nếu BE đổi role mà FE không login lại → role cũ. Chấp nhận cho MVP; nâng cấp bằng poll /me định kỳ nếu cần.
